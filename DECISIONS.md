## 2026-07-10 — Версии внешних зависимостей (SWIFTMIG-6)

| Зависимость | Версия | Пин |
|---|---|---|
| swiftlang/swift-syntax | 603.0.2 | `exact` |
| tuist/XcodeProj | 9.13.0 | `from` (SemVer, мажор 9) |
| apple/swift-argument-parser | 1.8.2 | `from` (SemVer, мажор 1) |

**Почему swift-syntax запинован точно (`exact`):** нумерация swift-syntax привязана к версии компилятора (серия 603.x.x ↔ Swift 6.3). Наш тулчейн — Swift 6.3.3 (Xcode 26.6), поэтому берём последний патч серии 603. Обновление тулчейна = осознанное обновление пина, а не тихий джамп через `from`, который может сломать парсинг новых синтаксических конструкций.

**Почему XcodeProj и argument-parser через `from`:** обычные SemVer-библиотеки, обратная совместимость в пределах мажора. Точные версии всё равно зафиксированы в `Package.resolved`, который закоммичен.

**Package.resolved коммитим** (убран из .gitignore): воспроизводимость сборки на CI и на других машинах — все получают ровно те же версии.

## 2026-07-10 — CLI на swift-argument-parser с первого дня (SWIFTMIG-6)

`main.swift` с ручным print заменён на `@main struct Migrator: ParsableCommand`. ArgumentParser бесплатно даёт `--help`, `--version` и структуру подкоманд, на которую в SWIFTMIG-9 лягут `analyze` и `report`. Альтернатива (ручной разбор CommandLine.arguments) отброшена: больше кода, хуже UX, нечего показать в главе «организация разработки».

## 2026-07-10 — Smoke-тесты на каждую внешнюю зависимость (SWIFTMIG-6)

Каждая внешняя зависимость покрыта минимальным тестом (парсинг строки в `SourceFileSyntax`; открытие пустого `.xcodeproj` из фикстуры; `--help` у CLI). Цель — ловить несовместимость версий при любом обновлении зависимостей или тулчейна сразу, на уровне `swift test`, а не в середине разработки фичи.


## 2026-07-10 — Доменная модель: value types, ссылки по именам, unified diff (SWIFTMIG-7)

Все core-сущности — структуры/enum'ы: `Sendable`, `Equatable`, `Codable`, без I/O-логики.

- **Зависимости модулей — по именам (`[String]`), а не по ссылкам на объекты.** Модель остаётся деревом без циклов: тривиально сериализуется (dry-run отчёты, кэш анализа), сравнивается в тестах. Сам граф с рёбрами-объектами строится поверх модели отдельным компонентом.
- **Патч фикса — unified diff строкой.** Единый формат для детерминированных фиксов, ответов LLM и превью в UI; человекочитаем и применим стандартными инструментами. Альтернатива (замена узла AST) отброшена: не работает для LLM-ответов и не показывается пользователю напрямую.
- **`MigrationRoute` — enum с associated value `deterministic(FixRuleID)`.** Маршрутизация «авто / LLM / вручную» выражена в типе: невозможно создать детерминированный маршрут без указания правила.
- **`FixRuleID` — типобезопасная обёртка над String**, а не голая строка: сигнатуры протоколов (SWIFTMIG-8) не позволят перепутать идентификатор правила с текстом.

## 2026-07-11 — Порты слоёв: ports & adapters, AST не протекает в контракты (SWIFTMIG-8)

Пять протоколов в `Sources/MigratorCore/Ports/`: ProjectLoader, DiagnosticsProvider, IssueClassifier, FixRule, LLMProvider. Ядро объявляет контракты, реализации приходят на этапах 1–3 через DI.

- **AST-контекст не входит в сигнатуру `IssueClassifier`.** Классификатор строит синтаксические деревья сам по путям из `SourceLocation`. Альтернатива (передавать `SourceFileSyntax` параметром) отброшена: тип swift-syntax в порте привязал бы все слои и моки к конкретной библиотеке парсинга.
- **`FixRule.makeFix` получает исходник строкой и возвращает `FixProposal?`.** Правило — чистая функция без I/O (чтение/запись файлов — забота вызывающего слоя), а `nil` — типизированное «не уверен» из пункта 12 плана: сомнение = отказ, случай уходит дальше по маршруту.
- **`DiagnosticsProvider` имеет default-реализацию** «проект = конкатенация модулей»: адаптеры реализуют один метод, но могут переопределить, если у системы сборки есть способ дешевле.
- **DTO LLM-слоя (`LLMFixRequest`/`LLMFixResponse`) — часть порта**, а не деталей реализации: структурированный ответ (патч + объяснение) фиксируется контрактом, провайдеры не могут вернуть сырой текст.

## 2026-07-11 — CLI: заглушка-лоадер за портом, коды возврата в одном месте (SWIFTMIG-9)

Команды `analyze`/`report` работают через `StubProjectLoader: ProjectLoader` (модули = подпапки Sources). Когда на этапе 1 появится настоящий SPM-лоадер, меняется одна строка DI — команды не трогаем: первая практическая выгода портов из SWIFTMIG-8.

- **Маппинг ошибок в коды возврата centralized в `CLISupport.loadProject`**: невалидный вход (нет пути/не проект) → 2, ошибка анализа → 1. Команды не знают про коды — не разъедутся между analyze и report.
- **`--json` печатает доменную модель как есть** (`AnalyzedProject` через Codable из SWIFTMIG-7) — отдельного слоя DTO для CLI нет, пока не понадобится.
- **Фикстура `Fixtures/DemoProject`** — общая для CLI-тестов и ручных прогонов; в её DemoCore уже лежит глобальная `var` — будущий тест-кейс для фиксов этапа 2.
