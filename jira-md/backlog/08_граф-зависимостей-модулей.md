# [SWIFTMIG-25] Граф зависимостей: построение, сортировка, порядок миграции
**Контекст:** Граф модулей — то, что превращает SwiftMigrator из анализатора файлов в инструмент планирования миграции: он отвечает на главный вопрос пользователя «с чего начать». Эпик строит граф из данных лоадеров (эпик 7), уточняет его реальными import-декларациями из AST (инфраструктура эпика 6), сортирует топологически и выдаёт волны миграции — какие модули мигрировать первыми и какие можно делать параллельно. Этот же граф — источник «соседних узлов» для экстрактора LLM-контекста (этап 3) и основа интерактивной визуализации в UI (этап 4), поэтому модель графа проектируется как публичное API MigratorCore, а не внутренняя утилита. На защите именно волны миграции — один из самых наглядных результатов работы.

## [SWIFTMIG-26] Модель графа и построение из структуры проекта
> estimate: 1д
> labels: core, graph

**Контекст:**
Базовая структура данных эпика: направленный граф «модуль → зависит от модуля», построенный из `AnalyzedProject`, который отдают лоадеры эпика 7. Модель должна быть value type и Sendable, как вся доменная модель, и достаточно общей, чтобы поверх неё работали сортировка, поиск циклов, волны и экспорт — без переделок. Внешние зависимости (SPM-пакеты) остаются в графе как узлы особого типа: мигрировать их нельзя, но для порядка и LLM-контекста они важны.

**Что сделать:**
- `ModuleGraph`: узлы (`ModuleNode`: имя, вид — internal/external/test) и рёбра «зависит от»
- Построение из `AnalyzedProject`: межтаргетные зависимости → рёбра; продуктовые/внешние зависимости → external-узлы
- Базовые запросы: зависимости модуля, обратные зависимости (кто зависит от меня), достижимость
- Инварианты с проверкой при построении: ребро не может вести в несуществующий узел; дубли рёбер схлопываются

**Технические заметки:**
- Представление — adjacency-списки поверх словаря `[ModuleID: Set<ModuleID>]`, плюс обратный индекс для reverse-запросов (понадобится для «кого разблокирует миграция модуля X»).
- ID модуля — имя таргета: в SPM оно уникально в пределах пакета; коллизии с именами external-продуктов разруливать префиксом вида узла. Записать соглашение в doc-комментарий.

```swift
struct ModuleGraph: Sendable, Codable {
    struct Node: Sendable, Codable { let id: ModuleID; let kind: NodeKind }
    // dependencies(of:), dependents(of:), allNodes, edges
}
```

**Критерии приёмки:**
- [ ] Граф DemoPackage (a→b→c) строится: 3 узла, 2 ребра, направление корректно
- [ ] Внешняя зависимость появляется как external-узел
- [ ] `dependents(of: c)` возвращает b (обратный индекс работает)

**Как проверить:**
`swift test --filter ModuleGraphTests`

**Вне скоупа:**
Рёбра из import-деклараций — следующая задача; здесь только манифестные зависимости.

## [SWIFTMIG-27] Уточнение рёбер по import-декларациям
> estimate: 1д
> labels: core, graph, parsing
> after: Модель графа и построение из структуры проекта

**Контекст:**
Манифест говорит, от чего модуль *может* зависеть, а import-декларации в коде — от чего он зависит *фактически*. Расхождения в обе стороны ценны: заявленная, но не импортируемая зависимость — кандидат на чистку и лишнее ребро в порядке миграции; import без манифестной зависимости (transitive import) — скрытая связь, которая всплывёт ошибками при миграции. Задача добавляет в граф второй слой рёбер и отчёт о расхождениях — это ещё и готовый материал в диплом про «инструмент находит проблемы структуры до начала миграции».

**Что сделать:**
- `ImportCollector` (визитор в инфраструктуре эпика 6): все import-декларации файла, включая `@testable` и подмодульные (`import Foo.Bar` → модуль `Foo`)
- Агрегация по модулю: множество фактически импортируемых модулей
- Наложение на граф: рёбра получают источник — `.manifest`, `.imports` или `.both`
- Отчёт расхождений: заявлено-но-не-импортируется / импортируется-без-объявления (для system-модулей типа Foundation — не расхождение, а external)

**Технические заметки:**
- Условные импорты внутри `#if canImport(...)` парсер видит во всех ветках — считаем ребром любую встреченную ветку (консервативно: лишнее ребро безопаснее пропущенного).
- Список system-модулей (Foundation, UIKit, Swift, os, ...) — константа с doc-комментарием; они всегда external и в расхождения не попадают.
- `@testable import` пометить флагом на ребре — тестовые связи в порядке миграции учитываются иначе (тесты мигрируют вместе со своим модулем).

**Критерии приёмки:**
- [ ] Фикстура с `import Foo.Bar` даёт ребро к модулю Foo
- [ ] Заявленная, но неиспользуемая зависимость попадает в отчёт расхождений
- [ ] Foundation не считается расхождением и помечен external

**Как проверить:**
`swift test --filter ImportGraphTests`

**Вне скоупа:**
Автоматическая чистка манифеста от лишних зависимостей — инструмент только сообщает; правки манифеста в scope диплома не входят.

## [SWIFTMIG-28] Топологическая сортировка и обнаружение циклов
> estimate: 1д
> labels: core, graph
> after: Уточнение рёбер по import-декларациям

**Контекст:**
Алгоритмическое ядро эпика. Сам алгоритм Кана — учебный, но два требования делают задачу содержательной: детерминизм (одинаковый вход → одинаковый порядок, иначе нестабильны тесты, JSON-вывод и текст диплома) и диагностика циклов с конкретным путём, а не булевым флагом. Манифестный граф SPM циклов содержать не может (SPM их запрещает), а вот граф по import-рёбрам и xcodeproj — могут; цикл — это не ошибка инструмента, а находка о проекте, которую надо показать пользователю.

**Что сделать:**
- Алгоритм Кана с лексикографическим tie-break: из доступных узлов всегда берётся минимальный по имени
- При недостижимости полного порядка — поиск и возврат конкретных циклов (список путей вида a→b→c→a)
- Режимы: сортировка по манифестным рёбрам (гарантированный DAG для SPM) и по объединённым — с отчётом циклов
- Property-тест: для случайных DAG порядок валиден (каждый узел после своих зависимостей) и стабилен между запусками

**Технические заметки:**
- Tie-break через отсортированную вставку в очередь достаточно: граф модулей маленький (десятки узлов), O(V·logV) не имеет значения — читаемость важнее.
- Поиск циклов — DFS по остаточному графу после Кана (узлы с ненулевой входящей степенью); вернуть каждый элементарный цикл один раз, нормализовав начало цикла минимальным именем.

**Критерии приёмки:**
- [ ] DemoPackage сортируется: c, b, a (зависимости раньше зависимых)
- [ ] Искусственный цикл в фикстуре обнаруживается, путь цикла возвращается полностью
- [ ] Два запуска на одном графе дают идентичный порядок (property-тест на 100 случайных DAG)

**Как проверить:**
`swift test --filter TopologicalSortTests`

**Вне скоупа:**
Разрыв циклов и рекомендации по рефакторингу — инструмент показывает цикл, решение за разработчиком.

## [SWIFTMIG-29] Волны миграции: порядок и параллелизм
> estimate: 1д
> labels: core, graph
> after: Топологическая сортировка и обнаружение циклов

**Контекст:**
Продуктовый смысл всего эпика: превратить порядок сортировки в план миграции. Правильная стратегия — от листьев к корню: сначала модули без внутренних зависимостей, чтобы каждый следующий модуль мигрировал, опираясь на уже строгие (Swift 6) зависимости, а не на `@preconcurrency`-заглушки. Модули одной волны независимы друг от друга — их можно мигрировать параллельно силами разных людей, что для команд и есть ответ на «непонятно, с чего начать» из проблематики диплома. Волны — центральный экран будущего UI и главный слайд защиты.

**Что сделать:**
- `MigrationPlan`: разбивка internal-узлов на волны — волна N содержит модули, все зависимости которых в волнах < N (external-узлы считаются волной 0 и в план не входят)
- Для каждого модуля в плане: волна, прямые зависимости, кого разблокирует (обратные зависимости)
- Модули из циклов — отдельная секция плана «требуют совместной миграции», с путём цикла
- Взвешивание волны количеством файлов модулей (грубая оценка трудоёмкости из `AnalyzedProject`)

**Технические заметки:**
- Волна = longest path от листьев (уровень узла = 1 + max(уровни зависимостей)); считается за один проход по топологическому порядку — переиспользовать результат предыдущей задачи, не пересортировывать.
- Тестовые модули в волны не включать, но в плане показать привязку «тесты модуля X мигрируют вместе с X».

**Критерии приёмки:**
- [ ] DemoPackage: волна 1 — c, волна 2 — b, волна 3 — a
- [ ] Ромбовидный граф (a→b, a→c, b→d, c→d): b и c в одной волне
- [ ] Модули цикла попадают в секцию совместной миграции, не теряются

**Как проверить:**
`swift test --filter MigrationPlanTests`

**Вне скоупа:**
Учёт количества ошибок конкурентности в весе волны — станет возможным после пункта 9 плана (диагностики); сейчас вес только по размеру.

## [SWIFTMIG-30] Экспорт графа и CLI-команда graph
> estimate: 1д
> labels: cli, graph
> after: Волны миграции: порядок и параллелизм

**Контекст:**
Выводит результат эпика наружу: команда `graph` печатает волны миграции в терминал и экспортирует граф в DOT и JSON. DOT-файл рендерится Graphviz'ом в картинку — это иллюстрации для текста диплома уже сейчас и промежуточный формат до интерактивного UI (этап 4, который возьмёт тот же JSON). Прогон на реальных проектах корпуса даст первые содержательные картинки «вот как выглядит структура настоящего проекта и его план миграции».

**Что сделать:**
- CLI-команда `graph <путь>`: текстовый вывод волн (волна → модули → счётчик файлов), секция циклов и расхождений манифест/импорты
- `--format dot`: экспорт в Graphviz DOT — external-узлы серым, волны как ранги, рёбра-расхождения пунктиром
- `--format json`: `ModuleGraph` + `MigrationPlan` одним документом, стабильный порядок ключей
- Прогон на 2–3 проектах корпуса, DOT-рендеры сохранить в `docs/samples/`

**Технические заметки:**
- DOT генерировать строками без внешних зависимостей (формат тривиален); имена модулей экранировать кавычками — дефисы и точки в именах ломают невалидный DOT.
- `rank=same` по волнам укладывает граф слоями — читаемость картинки для диплома важнее красоты; force-layout остаётся UI-этапу.

**Критерии приёмки:**
- [ ] `swift run migrator graph Tests/Fixtures/DemoPackage` печатает 3 волны
- [ ] DOT-вывод рендерится Graphviz без ошибок (`dot -Tsvg` в тесте, если dot доступен, иначе валидация синтаксиса)
- [ ] JSON стабилен между запусками; минимум два реальных графа сохранены в docs/samples/

**Как проверить:**
`swift run migrator graph Tests/Fixtures/DemoPackage --format dot`

**Вне скоупа:**
Интерактивная визуализация с кликами и индикацией статуса — этап 4 плана (UI); здесь только статический экспорт.
