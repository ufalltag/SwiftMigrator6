# [SWIFTMIG-19] Извлечение структуры проекта: SPM и xcodeproj
**Контекст:** До этого эпика SwiftMigrator умеет анализировать файлы, но не понимает, что такое «проект»: какие в нём модули, какие файлы кому принадлежат и кто от кого зависит. Этот эпик реализует `ProjectLoader` из эпика 5 для двух миров — SPM (полноценно, это scope MVP) и xcodeproj (best effort через библиотеку XcodeProj). Результат: путь к проекту на входе → заполненный `AnalyzedProject` с модулями и их файлами на выходе, а CLI-команда `analyze` начинает работать с реальными проектами, а не списком файлов. От качества этого слоя зависит корректность графа зависимостей (пункт 8 плана) и маппинг диагностик на модули (пункт 9) — ошибка здесь означает, что весь дальнейший анализ смотрит не на те файлы.

## [SWIFTMIG-20] Обёртка запуска внешних процессов
> estimate: 1д
> labels: core, infra

**Контекст:**
Извлечение структуры SPM-проекта делается через вызов `swift package`, а впереди ещё `swift build` и `xcodebuild` для сбора диагностик (пункт 9 плана). Всем им нужна одна и та же инфраструктура: асинхронный запуск процесса, захват stdout/stderr, таймаут и внятные ошибки. Пишем её один раз здесь, чтобы следующие эпики просто переиспользовали. Кривая работа с Process — классический источник зависаний (переполненный pipe buffer), поэтому задача идёт первой и покрывается тестами тщательнее остальных.

**Что сделать:**
- `ProcessRunner` в MigratorCore: команда + аргументы + рабочая директория → `ProcessResult` (exitCode, stdout, stderr)
- Асинхронный API, конкурентное чтение stdout/stderr (чтобы не встать на заполненном pipe)
- Таймаут с завершением процесса и доменной ошибкой
- Ошибки различимы: бинарь не найден / ненулевой exit / таймаут

**Технические заметки:**
- Foundation `Process` + `Pipe`: читать оба потока параллельно до `waitUntilExit`, иначе процесс с объёмным выводом (а у `xcodebuild` он огромный) навсегда зависнет на записи в переполненный буфер.
- `Process` не `Sendable` — держать его внутри изолированного контекста, наружу отдавать только `ProcessResult` (value type).

```swift
struct ProcessResult: Sendable {
    let exitCode: Int32
    let stdout: String
    let stderr: String
}

protocol ProcessRunning: Sendable {
    func run(_ executable: String, arguments: [String],
             workingDirectory: URL?, timeout: Duration) async throws -> ProcessResult
}
```

**Критерии приёмки:**
- [ ] `echo`-тест: stdout захватывается, exitCode 0
- [ ] Процесс, пишущий мегабайты в stdout, не зависает
- [ ] Таймаут срабатывает и завершает процесс (тест на `sleep`)
- [ ] Несуществующий бинарь даёт доменную ошибку, не крэш

**Как проверить:**
`swift test --filter ProcessRunnerTests`

**Вне скоупа:**
Стриминг вывода в реальном времени (прогресс-бары) — не нужен до UI-этапа; сейчас достаточно результата целиком.

## [SWIFTMIG-21] SPMProjectLoader: структура из Package.swift
> estimate: 2д
> labels: core, project
> after: Обёртка запуска внешних процессов

**Контекст:**
Главная задача эпика: SPM — основной формат проектов в scope диплома, на нём пойдут корпус этапа 0 и все измерения. Манифест нельзя парсить текстом — Package.swift это исполняемый Swift-код, поэтому структуру отдаёт сам тулчейн через `swift package describe --type json`. Loader превращает этот JSON в `AnalyzedProject`: таргеты → модули, их зависимости и списки исходников. Точность маппинга «файл → модуль» критична: на ней держится привязка диагностик и очерёдность миграции.

**Что сделать:**
- `SPMProjectLoader: ProjectLoader`: запуск `swift package describe --type json` через ProcessRunner, декодинг JSON → `AnalyzedProject`
- Маппинг: только Swift-таргеты (regular + test), их межтаргетные зависимости по именам, продуктовые и внешние зависимости помечаются как внешние
- Список файлов таргета: из `sources` в describe-выводе (тулчейн сам разрешает кастомные `path`/`exclude` — не дублировать эту логику руками)
- Фикстура `Tests/Fixtures/DemoPackage`: 3 таргета с зависимостями a→b→c, кастомным path у одного таргета и тест-таргетом
- Ошибки: не SPM-проект / битый манифест → доменная ошибка с текстом от тулчейна

**Технические заметки:**
- `describe --type json` запускает резолв манифеста и может ходить в сеть за зависимостями при первом вызове — в тестах фикстура должна быть без внешних зависимостей, чтобы тесты работали офлайн и быстро.
- JSON-схема describe отличается между версиями тулчейна — декодировать через свои DTO с опциональными полями, а не жёсткой структурой; версию тулчейна фиксируем в DECISIONS.md.
- C/Objective-C таргеты пропускаем с пометкой в результате (scope диплома — Swift), но их наличие не должно ронять загрузку.

**Критерии приёмки:**
- [ ] DemoPackage разбирается: 3 модуля, зависимости a→b→c корректны
- [ ] Кастомный `path` таргета учтён — файлы найдены
- [ ] Файлы тест-таргета не попадают в regular-модуль
- [ ] Папка без Package.swift даёт понятную доменную ошибку

**Как проверить:**
`swift test --filter SPMProjectLoaderTests`

**Вне скоупа:**
Построение полного графа с топологической сортировкой — пункт 8 плана; здесь только сырые связи «модуль → имена зависимостей» в модели.

## [SWIFTMIG-22] XcodeProjectLoader: базовая поддержка xcodeproj
> estimate: 2д
> labels: core, project
> after: SPMProjectLoader: структура из Package.swift

**Контекст:**
Реальные коммерческие приложения чаще живут в xcodeproj, чем в чистом SPM, поэтому хотя бы базовая поддержка сильно расширяет применимость инструмента (и это готовый ответ на вопрос комиссии «а на настоящем проекте?»). Формат сложный и местами недокументированный, поэтому статус задачи — best effort: покрываем типовой случай (native targets с Swift-файлами), явно фиксируем, что не поддержано. Задача идёт после SPM-лоадера, чтобы переиспользовать сложившиеся паттерны и тестовую инфраструктуру.

**Что сделать:**
- `XcodeProjectLoader: ProjectLoader` на библиотеке XcodeProj: native targets → модули, файлы из source build phase, зависимости из target dependencies
- Разрешение путей файлов относительно `sourceTree` (group/absolute/SOURCE_ROOT)
- Пропуск с пометкой: не-Swift таргеты, агрегатные таргеты, таргеты без исходников
- Фикстура: минимальный xcodeproj с двумя таргетами и зависимостью (собрать в Xcode и закоммитить)
- Неподдержанные конструкции собираются в `warnings` результата, не роняют загрузку

**Технические заметки:**
- В XcodeProj файлы таргета: `target.sourcesBuildPhase()?.files` → `PBXBuildFile.file` → `fullPath(sourceRoot:)`. Файлы, добавленные через folder references и новые buildable folders (Xcode 16), могут не разрешиться — это известное ограничение, фиксируем в warnings и DECISIONS.md.
- SPM-зависимости внутри xcodeproj (`XCSwiftPackageProductDependency`) в MVP помечаем как внешние модули без раскрытия их исходников.
- Workspace (`.xcworkspace`) не поддерживаем — только одиночный `.xcodeproj`; про workspace честно писать в ошибке.

**Критерии приёмки:**
- [ ] Фикстурный xcodeproj разбирается: 2 модуля, зависимость между ними, файлы найдены
- [ ] Не-Swift таргет пропущен с warning, загрузка не падает
- [ ] `.xcworkspace` на входе даёт понятную ошибку «не поддерживается, укажите .xcodeproj»

**Как проверить:**
`swift test --filter XcodeProjectLoaderTests`

**Вне скоупа:**
Workspace, CocoaPods, раскрытие SPM-зависимостей внутри xcodeproj, кастомные build settings — за рамками MVP (DECISIONS.md).

## [SWIFTMIG-23] Автоопределение типа проекта и фабрика лоадеров
> estimate: 0.5д
> labels: core, project
> after: XcodeProjectLoader: базовая поддержка xcodeproj

**Контекст:**
Пользователь указывает папку — инструмент сам понимает, SPM это или xcodeproj, и выбирает лоадер. Мелкая задача, но она определяет UX первого контакта с инструментом и снимает вопрос «какой флаг передать». Здесь же решается спорный случай «в папке и Package.swift, и .xcodeproj» — правило должно быть детерминированным и задокументированным.

**Что сделать:**
- `ProjectLoaderFactory`: папка → подходящий `ProjectLoader` или ошибка «проект не распознан»
- Правило приоритета: `Package.swift` в корне → SPM; иначе единственный `*.xcodeproj` → Xcode; оба присутствуют → SPM + warning (правило в DECISIONS.md)
- Несколько `.xcodeproj` в папке → ошибка со списком найденных и просьбой указать конкретный

**Технические заметки:**
- Фабрика — единственное место, знающее о конкретных лоадерах; весь остальной код работает с протоколом `ProjectLoader`. Это тот же принцип подмены, что у `LLMProvider`, — пригодится для мок-лоадера в UI-тестах.

**Критерии приёмки:**
- [ ] Папка DemoPackage → SPMProjectLoader
- [ ] Папка с фикстурным xcodeproj → XcodeProjectLoader
- [ ] Пустая папка → ошибка «проект не распознан» с подсказкой

**Как проверить:**
`swift test --filter ProjectLoaderFactoryTests`

**Вне скоупа:**
Интерактивный выбор проекта в UI — этап 4 плана.

## [SWIFTMIG-24] Интеграция в CLI: analyze по реальному проекту
> estimate: 1д
> labels: cli
> after: Автоопределение типа проекта и фабрика лоадеров

**Контекст:**
Собирает эпик в демонстрируемый результат: `migrator analyze <путь>` теперь принимает настоящий проект, сам определяет тип, загружает структуру и прогоняет анализ эпика 6 по каждому модулю. Это первая точка, где инструмент можно запустить на проектах корпуса из этапа 0 и получить осмысленный вывод — фактически ранний прототип dry-run отчёта (пункт 22 плана). Вывод этой команды пойдёт в диплом как первые эмпирические данные.

**Что сделать:**
- `analyze` принимает путь к проекту: фабрика → лоадер → `AnalyzedProject` → пофайловый анализ (ModuleAnalyzer из эпика 6) по каждому модулю
- Вывод: таблица по модулям — файлов, типов, UIKit-наследников, глобальных `var`; внизу сумма и время
- `--json`: полная структура проекта + индексы деклараций, стабильный порядок
- Прогнать на 2–3 реальных проектах из корпуса этапа 0, вывод сохранить в `docs/samples/`

**Технические заметки:**
- Модули анализировать в порядке из лоадера, файлы внутри модуля — отсортированными: детерминизм вывода уже требование (тесты сравнивают JSON).
- Ошибка загрузки одного модуля не должна убивать весь анализ — модуль помечается failed, остальные анализируются (устойчивость понадобится на больших проектах корпуса).

**Критерии приёмки:**
- [ ] `swift run migrator analyze Tests/Fixtures/DemoPackage` печатает таблицу по 3 модулям
- [ ] Тот же вызов с `--json` даёт стабильный между запусками вывод
- [ ] Минимум два реальных проекта из корпуса проанализированы, вывод сохранён в docs/samples/

**Как проверить:**
`swift run migrator analyze Tests/Fixtures/DemoPackage --json`

**Вне скоупа:**
Граф зависимостей и очерёдность миграции в выводе — пункт 8 плана, следующий эпик; счётчики ошибок компилятора — пункт 9.
