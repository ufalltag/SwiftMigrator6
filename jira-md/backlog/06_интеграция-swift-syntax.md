# [SWIFTMIG-13] Интеграция swift-syntax: парсинг, обход, трансформации
**Контекст:** swift-syntax — фундамент детерминированного слоя SwiftMigrator: на нём строятся анализ кода, классификация ошибок и все автоматические фиксы. Этот эпик превращает библиотеку из подключённой зависимости (SWIFTMIG эпик 5) в рабочий инструментарий: чтение файлов в AST, инфраструктура визиторов, первая трансформация и привязка позиций для маппинга диагностик компилятора. Результат эпика — прототип полного пайплайна "файл → AST → анализ → трансформация → файл", на который лягут все FixRule этапа 2. Ошибки в дизайне этого слоя дороги: его API будут использовать классификатор, экстрактор контекста для LLM и каждое правило фикса.

## [SWIFTMIG-14] Обёртка чтения исходников в AST
> estimate: 1д
> labels: core, parsing

**Контекст:**
Точка входа всего анализа: превращение файла с диска в `SourceFileSyntax`. Обёртка изолирует остальной код от прямых вызовов swift-syntax, чтобы при обновлении библиотеки (а она меняет API между мажорными версиями) правки были в одном месте. Также здесь решается вопрос кодировок, битых файлов и генерируемого кода, который парсить не нужно. Все последующие задачи эпика строятся на этой обёртке.

**Что сделать:**
- Тип `ParsedSource`: путь + `SourceFileSyntax` + исходный текст
- Сервис `SourceParser` в MigratorCore: файл/строка → `ParsedSource`
- Обработка ошибок: нечитаемый файл, невалидный UTF-8 — свой тип ошибки, не крэш
- Фильтр служебных файлов: `.build/`, `DerivedData/`, `*.generated.swift` — пропускаются
- Фикстуры: папка `Tests/Fixtures/Sources/` с 3–4 эталонными swift-файлами

**Технические заметки:**
- Парсер swift-syntax не бросает ошибки на невалидном Swift — он строит дерево с missing/unexpected узлами. Проверка "файл распарсился" ≠ "файл валиден"; на этапе анализа это норм, валидность подтверждает компилятор.
- Читать файлы как `String`, парсить через `Parser.parse(source:)`.

```swift
struct ParsedSource: Sendable {
    let path: URL
    let text: String
    let syntax: SourceFileSyntax
}

protocol SourceParsing: Sendable {
    func parse(fileAt path: URL) throws -> ParsedSource
    func parse(text: String, path: URL) -> ParsedSource
}
```

**Критерии приёмки:**
- [ ] Фикстурные файлы парсятся, дерево содержит ожидаемые декларации
- [ ] Нечитаемый файл даёт доменную ошибку с путём в описании
- [ ] Файлы из `.build/` отфильтровываются

**Как проверить:**
`swift test --filter SourceParserTests`

**Вне скоупа:**
Обход всего проекта и параллельный парсинг множества файлов — это задача «Пакетный анализ модуля» ниже.

## [SWIFTMIG-15] Инфраструктура визиторов и сбор деклараций
> estimate: 2д
> labels: core, parsing
> after: Обёртка чтения исходников в AST

**Контекст:**
Визиторы — способ извлекать факты из AST: какие типы объявлены, кто наследует UIKit-классы, где глобальные переменные. Эти факты нужны и классификатору (этап 2), и экстрактору контекста для LLM (этап 3), поэтому собираем их в единую модель `DeclarationIndex`, а не разбрасываем ad-hoc обходы по коду. Задача формирует паттерн, по которому будут писаться все последующие анализаторы. Это самая переиспользуемая часть эпика.

**Что сделать:**
- Базовый паттерн на `SyntaxVisitor`: один визитор = один тип фактов
- Визиторы первой очереди: `TypeDeclCollector` (классы/структуры/enum + наследование), `GlobalVarCollector` (глобальные и статические `var`), `ClosureCollector` (замыкания с их контекстом)
- Модель `DeclarationIndex`: результаты всех визиторов по файлу, с позициями
- Композитный прогон: один проход по дереву, несколько коллекторов

**Технические заметки:**
- `SyntaxVisitor` требует указать `viewMode:` — использовать `.sourceAccurate`.
- Наследование от UIKit определяем по имени в inheritance clause (`UIViewController` и т.п.) — на уровне синтаксиса семантики нет, это осознанное ограничение слоя (без SourceKit). Список UIKit-баз — константа, задокументировать.
- Позиции узлов брать через `node.positionAfterSkippingLeadingTrivia`, иначе локации будут указывать на комментарии перед декларацией.

```swift
struct DeclarationIndex: Sendable {
    let types: [TypeDeclInfo]        // имя, вид, наследование, позиция
    let globalVars: [GlobalVarInfo]
    let closures: [ClosureInfo]
}
```

**Критерии приёмки:**
- [ ] На фикстуре с известным содержимым индекс совпадает с эталоном
- [ ] Класс, наследующий `UIViewController`, помечается как UIKit-наследник
- [ ] Один проход по дереву наполняет все коллекторы (проверено счётчиком обходов)

**Как проверить:**
`swift test --filter DeclarationIndexTests`

**Вне скоупа:**
Семантический анализ (реальное разрешение типов через SourceKit/индекс-стор) — осознанно не делаем, фиксируем как ограничение в DECISIONS.md.

## [SWIFTMIG-16] Первая трансформация: прототип FixRule на SyntaxRewriter
> estimate: 2д
> labels: core, fixes
> after: Инфраструктура визиторов и сбор деклараций

**Контекст:**
Первая настоящая модификация кода — прототип, который задаст форму всем правилам фиксов этапа 2. Берём самый простой содержательный случай: глобальный `var`, которому нигде нет присваивания, превращается в `let` (устраняет ошибку shared mutable state). На этой задаче проверяется весь контракт `FixRule` из эпика 5: применимость → трансформация → сохранение форматирования. Если паттерн окажется неудобным, дешевле поменять его сейчас, чем после десяти написанных правил.

**Что сделать:**
- Реализация `GlobalVarToLetRule: FixRule`: проверка применимости (нет присваиваний после инициализации в этом файле) + трансформация через `SyntaxRewriter`
- Сохранение trivia: комментарии и отступы вокруг изменённого узла не теряются
- Результат — новый текст файла + структура `AppliedFix` (что, где, каким правилом)
- Снапшот-тесты: файл «до» и эталон «после» лежат рядом в фикстурах

**Технические заметки:**
- В swift-syntax узлы иммутабельны: rewriter возвращает новое дерево, текст получаем через `description`.
- Проверка «нет присваиваний» на уровне одного файла — честное ограничение прототипа: кросс-файловые присваивания увидит только компилятор при верификации (этап 2, пункт 13 плана). Записать в DECISIONS.md.
- Правило обязано уметь отказаться: сомнение → `notApplicable`, а не рискованный фикс. Это принцип всех будущих правил (план, пункт 12).

**Критерии приёмки:**
- [ ] Снапшот-тест: `var x = 1` (без присваиваний) → `let x = 1`, форматирование сохранено
- [ ] Негативный тест: `var` с присваиванием ниже по файлу — правило отвечает `notApplicable`
- [ ] `AppliedFix` содержит позицию и ID правила

**Как проверить:**
`swift test --filter GlobalVarToLetRuleTests`

**Вне скоупа:**
Остальные правила фиксов (`@MainActor`, `Sendable`, `@preconcurrency`) — это этап 2 плана, отдельный эпик.

## [SWIFTMIG-17] Привязка позиций: SourceLocation для диагностик
> estimate: 1д
> labels: core, parsing
> after: Обёртка чтения исходников в AST

**Контекст:**
Компилятор сообщает об ошибках в координатах «файл:строка:колонка», а swift-syntax работает в абсолютных смещениях по дереву. Чтобы связать диагностику Swift 6 с конкретным узлом AST (а это ядро классификатора, пункт 10 плана), нужен двусторонний конвертер координат. Без него весь пайплайн «ошибка компилятора → узел → фикс» не собирается. Задача маленькая, но блокирует интеграцию с диагностиками (пункт 9 плана).

**Что сделать:**
- Обёртка над `SourceLocationConverter`: позиция узла → строка/колонка и обратно
- Поиск узла по локации диагностики: строка/колонка → минимальный охватывающий узел AST
- Тесты на граничные случаи: первая строка, последняя строка, многобайтовые символы (кириллица в строковых литералах, эмодзи)

**Технические заметки:**
- `SourceLocationConverter` создаётся на файл и переиспользуется — создание на каждый запрос дорого.
- Колонки у компилятора 1-based и считаются в UTF-8 байтах — сверить с тем, что отдаёт converter, и зафиксировать соглашение в doc-комментарии (классическое место off-by-one багов).

**Критерии приёмки:**
- [ ] Round-trip тест: позиция узла → строка/колонка → тот же узел
- [ ] Файл с кириллицей и эмодзи в литералах конвертируется без смещений
- [ ] Поиск по локации возвращает минимальный охватывающий узел

**Как проверить:**
`swift test --filter SourceLocationTests`

**Вне скоупа:**
Сам парсинг вывода компилятора (`-serialize-diagnostics`) — пункт 9 плана, следующий эпик.

## [SWIFTMIG-18] Пакетный анализ модуля и CLI-команда
> estimate: 1д
> labels: core, cli
> after: Инфраструктура визиторов и сбор деклараций

**Контекст:**
Собирает задачи эпика в видимый результат: прогон всех визиторов по всем файлам модуля с выводом сводки. Это первое «живое» использование инструмента и заготовка для dry-run отчёта (пункт 22 плана). Через эту команду удобно проверять инструмент на корпусе проектов этапа 0 уже сейчас, до появления классификатора. Плюс это демо-материал: скриншот вывода пойдёт в текст диплома.

**Что сделать:**
- `ModuleAnalyzer` в MigratorCore: список файлов → параллельный парсинг (TaskGroup) → суммарный `DeclarationIndex` по модулю
- Оживить CLI-команду `analyze` из эпика 5: вместо мок-данных — реальный анализ: количество файлов, типов, UIKit-наследников, глобальных `var`
- Флаг `--json` выдаёт индекс в машиночитаемом виде
- Замер времени анализа в выводе (пригодится для раздела производительности)

**Технические заметки:**
- Параллелить через `withThrowingTaskGroup`, результат сливать детерминированно (сортировка по пути файла) — иначе JSON-вывод будет нестабилен между запусками и сломает сравнение в тестах.
- `SourceFileSyntax` не `Sendable` — из таска наружу отдавать уже собранный `DeclarationIndex` (value type), а не дерево. Это первая практическая проверка нашей же доменной модели на Swift 6 concurrency.

**Критерии приёмки:**
- [ ] `swift run migrator analyze Tests/Fixtures/DemoProject` печатает реальную сводку
- [ ] Повторные запуски дают идентичный `--json` вывод
- [ ] Анализ фикстурного проекта укладывается в секунды, время печатается

**Как проверить:**
`swift run migrator analyze Tests/Fixtures/DemoProject --json`

**Вне скоупа:**
Извлечение структуры проекта из Package.swift (какие файлы в каком модуле) — пункт 7 плана; здесь список файлов передаётся явно или берётся обходом папки.
