# PROTOCOL — формат epic-task.md для выгрузки в Jira

Структура папки:

```
jira-md/
├── PROTOCOL.md      ← этот файл: формат + шаблон описания задачи
├── jira_sync.py     ← единственный скрипт (md → Jira wiki, связи, DoD)
├── Makefile         ← make jira_dry / make jira_tasks / make backlog
├── dod.md           ← общий Definition of Done, добавляется к каждой задаче
├── epic-task.md     ← рабочий файл: сюда нейронка пишет задачи по шаблону ниже
└── backlog/         ← история миграций, один md-файл на выгрузку
```

---

## 1. Разметка

**Первая строка — шапка (обязательна):**

```
--- <номер пункта плана> <короткий тезис>
```

Из неё собирается имя файла в backlog/: `05_каркас-проекта.md`.

**Маркеры:**

| Маркер | Значение |
|---|---|
| `# Название` | Эпик (всё до следующего `#` — его) |
| `# @SWIFTMIG-1 Название` | Существующий эпик, задачи добавятся в него |
| `## Название` | Задача внутри текущего эпика |
| `## [Bug] Название` | Задача с типом: `[Task]` (default), `[Bug]`, `[Story]` |
| `> estimate: 2д` | Оценка (метаполя — сразу после заголовка задачи) |
| `> labels: core, arch` | Лейблы через запятую |
| `> after: <ключ или название>` | Эта задача блокируется другой (создаст связь в Jira) |
| `> blocks: <ключ или название>` | Эта задача блокирует другую |
| `<!-- ... -->` | Комментарий, в Jira не попадает |

В `after`/`blocks` можно указать существующий ключ (`SWIFTMIG-4`) или точное
название задачи из этого же файла — скрипт свяжет их после создания.

**Markdown в описаниях** конвертируется в Jira wiki markup автоматически:
`**жирный**`, `` `код` ``, блоки ```swift, списки, `### заголовки`, `- [ ]` чекбоксы,
ссылки `[текст](url)`. Пиши обычный markdown — в Jira будет красиво.

---

## 2. Шаблон описания задачи

Каждая задача пишется по этой структуре (секции — жирным, порядок фиксированный):

```markdown
## Название задачи
> estimate: 2д
> labels: core, arch
> after: Создать структуру репозитория и таргеты

**Контекст:**
3–4 предложения: какую роль задача играет в проекте, что от неё зависит,
почему она устроена именно так и что будет, если сделать её плохо.
Контекст — это то, что забывается через месяц; здесь его сохраняем.

**Что сделать:**
- конкретные действия списком
- каждое действие проверяемо

**Технические заметки:**
Структурированные и важные: выбранные библиотеки/API, известные грабли,
ссылки на доку. Якорные сигнатуры и типы — кодом:

```swift
protocol ProjectLoader: Sendable {
    func load(at path: URL) async throws -> AnalyzedProject
}
```

Реализацию в задачу не пишем — только контракты и решения,
которые уже приняты осознанно.

**Критерии приёмки:**
- [ ] измеримый результат 1
- [ ] измеримый результат 2

**Как проверить:**
Конкретная команда или сценарий:
`swift test --filter DomainModelTests`

**Вне скоупа:**
Что сознательно НЕ делаем в этой задаче и где это будет сделано.
```

Правила к секциям:
- **Контекст** — обязателен, 3–4 предложения, без воды.
- **Технические заметки** — только принятые решения и грабли; сниппеты для
  якорных типов/сигнатур, не для реализации.
- **Критерии приёмки** — чекбоксы, каждый проверяем бинарно (да/нет).
- **Как проверить** — команда, которую можно скопировать и запустить.
- **Вне скоупа** — обязателен, минимум одна строка; защита от расползания задачи.

---

## 3. Definition of Done (общий)

Лежит в `dod.md` и автоматически дописывается в конец каждой задачи при
выгрузке. В самих задачах его не повторяем. Меняется в одном месте.

---

## 4. Правила валидации

1. Нет шапки `---` → ошибка.
2. Задача вне эпика → ошибка с номером строки.
3. Задача без описания → warning.
4. `after`/`blocks` ссылается на несуществующее название → ошибка до выгрузки.

## 5. Жизненный цикл

```
нейронка пишет в epic-task.md по шаблону
  → ты смотришь глазами → make jira_dry
  → make jira_tasks
  → задачи в Jira (описание в wiki markup + DoD + связи blocks/after)
  → архив в backlog/NN_тезис.md с ключами в заголовках
  → epic-task.md очищается
```
