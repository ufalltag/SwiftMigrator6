# SwiftMigrator — план разработки дипломной работы

Инструмент автоматизированной миграции iOS/macOS проектов с Swift 5 на Swift 6.

Общий срок: ~14–17 недель при параллельном написании текста.

---

## Этап 0. Подготовка и исследование (1–2 недели)

1. **Собрать корпус тестовых проектов.** 5–10 open-source iOS-проектов на Swift 5 разного размера (маленький ~5К строк, средний ~30К, большой 100К+). Зафиксировать коммиты — это неизменный бенчмарк для всех измерений.
2. **Составить каталог ошибок.** Прогнать каждый проект с `-strict-concurrency=complete`, собрать все диагностики компилятора в таблицу: текст ошибки, код диагностики, файл, частота. Эмпирическая база для диплома.
3. **Классифицировать вручную.** Для каждой категории пометить: детерминированно исправима / нужен контекст (LLM) / только вручную. Таблица станет спецификацией классификатора и главой диплома.
4. **Изучить прецеденты.** Xcode fix-its, `swift-format`, апплевский Swift Migrator (Swift 3→4). Обзор аналогов — обязательная глава, писать пока свежо.

## Этап 1. Фундамент: парсинг и анализ (2–3 недели)

5. **Настроить проект.** macOS SwiftUI + SPM, отдельные таргеты: `MigratorCore` (логика без UI), `MigratorUI`, `MigratorCLI`. Разделение core/UI позволяет тестировать логику без запуска приложения.
6. **Интеграция swift-syntax.** Парсинг файла в дерево, обход визиторами, простейшие трансформации (например, поиск всех глобальных `var`). Начать с игрушечных примеров.
7. **Извлечение структуры проекта.** Парсинг `Package.swift` для SPM; для xcodeproj — библиотека XcodeProj (tuist). MVP: полная поддержка SPM, xcodeproj — по возможности. Зафиксировать ограничение письменно.
8. **Построение графа зависимостей.** Модули → узлы, import'ы → рёбра. Топологическая сортировка + обработка циклов. Сразу покрыть юнит-тестами.
9. **Сбор диагностик компилятора.** Запуск `swift build`/`xcodebuild` с флагами строгой конкурентности, парсинг вывода (через `-serialize-diagnostics` или JSON, не regex по тексту). Связка: диагностика → узел графа → категория ошибки.

## Этап 2. Детерминированный слой (3–4 недели)

10. **Классификатор ошибок.** По кодам диагностик + анализу AST определять категорию и маршрут (авто / LLM / manual). Отдельный модуль с таблицей правил — легко тестировать и описывать.
11. **Реализация фиксов по одному, в порядке частотности.** Для каждого: правило применимости, трансформация AST, юнит-тесты «до/после» на снапшотах кода. Базовый набор:
    - `@MainActor` для UIKit-наследников
    - `Sendable` для immutable structs
    - глобальные `var` → `let`
    - `@preconcurrency import`
12. **Правила безопасности фиксов.** Каждый фикс должен уметь сказать «не уверен» и передать случай дальше. Лучше меньше покрытие, но нулевая поломка кода.
13. **Верификация.** После пакета фиксов — перекомпиляция модуля. Ошибок меньше и нет новых → фиксы приняты, иначе откат. Откат через Git (отдельная ветка/коммит на пакет изменений).

## Этап 3. LLM-слой (2–3 недели)

14. **Экстрактор контекста.** Для ошибки собрать: объявление типа, сигнатуры мест использования, объявления смежных типов из графа. Замерять размер контекста в токенах — цифры для слайда («1.2К токенов на ошибку против 400К на весь проект»).
15. **Промпт и формат ответа.** Структурированный ответ (JSON с патчем или заменой узла AST). Валидация: применился ли патч, парсится ли, компилируется ли.
16. **Diff-превью и подтверждение.** Пользователь видит изменение, принимает/отклоняет. Отклонённые случаи логируются — данные для раздела «анализ ограничений».
17. **Обработка отказов модели.** Retry с уточнением, лимит попыток, деградация в «ручной» статус. Готовый механизм на вопрос «а что если AI ошибётся».
18. **Абстракция провайдера.** Протокол `LLMProvider` — независимость от конкретного API + мок-провайдер для тестов и офлайн-демо.

## Этап 4. UI и UX (2–3 недели)

19. **Дашборд миграции.** Сводка «всего ошибок / авто / AI / вручную», прогресс по модулям. Первый экран демо.
20. **Интерактивный граф модулей.** Цветовая индикация (мигрирован / в процессе / проблемный), клик по узлу → список ошибок. Не закапываться в рендеринг: force-directed layout или иерархия по уровням топологической сортировки.
21. **Экран работы с ошибкой.** Код, объяснение проблемы, предложенный фикс, diff, кнопки принять/отклонить.
22. **Dry-run отчёт.** Экспорт анализа в HTML/Markdown без внесения изменений. Отдельная ценность + запасной вариант демо.

## Этап 5. Эксперимент и измерения (2 недели — не сжимать!)

23. **Прогон на всём корпусе.** Для каждого проекта фиксировать: ошибок до, исправлено детерминированно, исправлено AI (принято/отклонено), осталось вручную, время работы, стоимость API-запросов.
24. **Проверка корректности.** Компиляция + прогон тестов проектов после миграции. Падающие тесты — разбор причин, материал для главы об ограничениях.
25. **Сравнение с baseline.** Тот же проект вручную (замерить своё время на модуле) или через Xcode fix-its. Цифра «инструмент — 20 минут, вручную — 6 часов» — главный аргумент защиты.
26. **Оформить негативные результаты.** Категории, где подход не сработал, — научная честность, а не провал.

## Этап 6. Текст диплома (параллельно, финализация 2–3 недели)

27. **Писать по ходу, а не в конце.** После каждого этапа — черновик соответствующей главы.
28. **Структура:** обзор проблемы и аналогов → исследование корпуса ошибок → архитектура → детерминированный слой → LLM-слой → эксперимент → результаты и ограничения.
29. **Все схемы — из реальных данных.** Диаграмма архитектуры, скриншот реального графа, таблицы с реальными цифрами.

## Этап 7. Подготовка к защите (последняя неделя)

30. **Демо с тройной страховкой:** живой прогон на маленьком проекте (отрепетированный) → записанное видео → скриншоты в слайдах. LLM-часть для живого демо — на мок-провайдере с закэшированными ответами.
31. **Репетиция с таймером.** 7–10 минут: проблема (1 мин) → подход (2 мин) → архитектура (2 мин) → демо/результаты (3 мин) → выводы (1 мин).
32. **Список ожидаемых вопросов с ответами** (2–3 предложения, желательно с цифрой):
    - Чем отличается от Xcode fix-its?
    - Что если AI сломает код?
    - Почему не отправить весь проект в модель?
    - Как проверяется корректность?
    - Почему такие scope-ограничения?
    - Сколько стоит миграция в API-токенах?
33. **Прогон перед научруком** за 1–2 недели до защиты.

---

## Общие принципы

- **Пункты 1–4 и 23–26 важнее красоты кода.** Дипломы валятся на «докажите, что это работает», а не на качестве классов. Корпус проектов и таблица измерений — броня.
- **Вести дневник решений** (markdown-файл): «выбрал X вместо Y, потому что...». Половина вопросов комиссии — «почему вы сделали так».
- **Если времени меньше** — резать ширину (меньше категорий фиксов, только SPM), но не этап эксперимента.
